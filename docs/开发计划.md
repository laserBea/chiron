## 一、需求分析总览

### 1.1 核心能力需求

| 能力 | 优先级 | 说明 |
|---|---|---|
| IntentTemplate 结构化声明 | P0 | Agent 生成标准化的意图模板，作为一致性校验的输入 |
| Calldata 解码 | P0 | 将原始 calldata 解码为结构化参数，支持主流协议 ABI |
| L1 确定性校验 | P0 | 6 条校验规则（合约/函数/参数/金额/异常/嵌套），本地运行 |
| 校验结果返回 | P0 | PASS / FAIL / UNCERTAIN + 人类可读原因 |
| 内置协议地址表 | P0 | 主流 DeFi 协议的合约地址 → Action 映射 |
| 链上存证 | P0 | VerificationReceipt 哈希写入 EVM 链 |
| 异步非阻塞设计 | P0 | 校验和存证均不阻塞交易签名和广播 |
| SDK 集成 API | P0 | 3 行代码集成的 JS/TS SDK |
| BondPool | P1 | Agent 质押代币，质押量决定交易上限 |
| CircuitBreaker | P1 | 连续违规/超日额度自动暂停 Agent |
| L2 Verifier 升级路径 | P1 | 多 AI 模型交叉验证（GPT + Claude + Gemini）|
| 钱包插件 | P2 | MetaMask Snap 展示验证结果 |
| IntentTemplateRegistry | P2 | 社区提交 Action → 合约映射 |
| 批量交易支持 | P2 | ERC-8211 batch 每步独立校验 |
| 挑战期 | P2 | 对历史验证结果发起争议 |
| 质押罚没 | P3 | 持续违规 → 部分质押分配给受害者 |
| 本地轻量 EVM 模型 | P3 | 替代 L2 LLM 依赖 |

### 1.2 质量需求

| 属性 | 目标值 |
|---|---|
| L1 校验延迟 | ≤ 15ms（p95）|
| L1 校验覆盖率 | ≥ 95%（剩余 5% 走 L2）|
| L1 准确率 | ≥ 99%（人工抽检验证）|
| 误报率 | ≤ 1% |
| 链上存证 Gas | ≤ 30,000 / 次 |
| 集成代码行 | ≤ 3 行新增 |
| 支持的 EVM 链 | ≥ 5 条 |
| 支持的 Action 类型 | ≥ 9 种（MVP）|
| 内置协议表覆盖 | ≥ 20 个主流合约 |

### 1.3 安全需求

| 需求 | 说明 |
|---|---|
| 校验结果防篡改 | VerificationReceipt 哈希上链存证 |
| 防重放 | Receipt 绑定 txHash + chainId |
| Verifier 签名 | L2 结果需 Verifier 私钥签名 |
| Intent 防篡改 | IntentTemplate 在 Agent 内存生成，不经过外部输入 |
| 速率限制 | 每个 Agent 日存证上限 1000 次 |

## 二、技术架构决策

### 2.1 技术栈选型

| 层 | 技术选型 | 理由 |
|---|---|---|
| SDK 语言 | TypeScript | EVM 生态主流语言，类型安全 |
| 合约框架 | Foundry | 最快的 Solidity 开发框架，原生 fuzz 测试 |
| 链上存储 | Solidity | EVM 原生 |
| L2 验证服务 | Node.js + Express | 轻量，与 TS SDK 共享类型 |
| LLM 调用 | OpenAI / Anthropic / Google SDK | 标准化 API |
| 钱包插件 | MetaMask Snaps | EVM 生态覆盖率最高 |
| CI/CD | GitHub Actions | 与 GitHub 仓库原生集成 |
| 包管理 | pnpm workspace | monorepo 管理 |

## 四、风险和依赖管理

### 4.1 外部依赖

| 依赖 | 用途 | 风险 | 缓解 |
|---|---|---|---|
| OpenAI / Anthropic / Google API | L2 AI 验证 | API 变更/费率上涨/断服 | 本地模型替代路径（P3）|
| Etherscan API | ABI 查找回退 | 限频/断服 | 内置 ABI 缓存 + 社区贡献 |
| MetaMask Snaps | 钱包插件 | Snap 权限限制 | 先做 Web UI 版本 |
| ERC-8211 标准 | 批量交易 | 标准仍在 Draft 阶段 | 先支持自定义 batch 格式 |

### 4.2 技术风险

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| Calldata 解码失败率高于预期 | 中 | L1 覆盖率下降 | 多级解码（4byte → ABI → AI 辅助）|
| L1 误报率高于 1% | 低 | 用户体验下降 | 可配置的宽松模式 + 用户反馈闭环 |
| L2 LLM 响应不稳定 | 中 | L2 验证不可用 | 超时跳过 + 本地轻量模型备选 |
| 新协议来不及加入地址表 | 高 | UNCERTAIN 率上升 | 社区提交机制 + 自动扫描工具 |

### 4.3 里程碑

```
M1 (Week 2)  ── ✔ Phase 0 完成：类型定义、ABI 索引、CI 就绪
M2 (Week 5)  ── ✔ Phase 1 完成：SDK 可本地运行 L1 校验
M3 (Week 8)  ── ✔ Phase 2 完成：合约部署测试网
M4 (Week 11) ── ✔ Phase 3 完成：15 种攻击测试通过，安全审计准备就绪
M5 (Week 15) ── ✔ Phase 4 完成：L2 Verifier 在线，MetaMask Snap 可用
M6 (Week 20) ── ✔ Phase 5 完成：5 链部署，npm 发布
```
