### 2.2 数据流

```
Agent 生成交易意图
      │
      ▼
IntentParser.create(action, params)     ← 结构化声明
      │
      ▼
TxDecoder.decode(target, data)          ← Calldata 解码
      │
      ▼
ConsistencyChecker.check(intent, tx)    ← 6 条确定性规则
  ├─ R1: 目标合约匹配协议表
  ├─ R2: 函数选择器匹配 Action
  ├─ R3: 代币参数匹配
  ├─ R4: 金额在滑点范围内
  ├─ R5: 无意外参数
  └─ R6: 无隐藏 delegatecall/CREATE2
      │
      ├── PASS ──→ Agent 签名 + 广播（~10ms，0 Gas）
      └── FAIL ──→ 阻止 + 返回详细原因
```

## 三、核心模块详解

### 3.1 types.ts — 类型定义

**用途：** 定义 Chiron 的所有数据结构，是其他模块的依赖基础。

| 类型 | 说明 |
|---|---|
| `ActionType` 枚举 | 9 种 Agent 操作类型：SWAP, APPROVE, BRIDGE, DEPOSIT, WITHDRAW, STAKE, VOTE, PERMIT, CUSTOM |
| `IntentTemplate` | 结构化意图声明：actionType, protocolHash, tokenIn/Out, amount, receiver, deadline |
| `DecodedTx` | 解码后的交易：target, selector, functionName, params, hasDelegateCall, hasCreate2 |
| `VerificationReceipt` | 验证结果：allowed, reason, intentHash, txHash, l1Result, details |
| `ChironConfig` | SDK 配置：chainId, securityLevel, tokenResolver |

### 3.2 registry-data.ts — 协议注册表

**用途：** 内置已知的协议合约地址 → Action 映射表，供 L1 校验器查询。

**覆盖范围（40 个协议，5 条链）：**

| 链 | 协议数 | 关键协议 |
|---|---|---|
| Ethereum | 26 | Uniswap, Aave, Morpho, Lido, EigenLayer, Cow Swap, Pendle, Sky/Maker... |
| Arbitrum | 5 | Uniswap, Aave, GMX, Camelot, Radiant |
| Optimism | 4 | Uniswap, Velodrome, Synthetix, Sonne |
| Base | 3 | Uniswap, Aerodrome, Seamless |
| Polygon | 2 | Uniswap, QuickSwap |

**数据来源：** DeFi Llama TVL 排名 + 手动采集合约地址。

**覆盖率（按加权 TVL 计算）：** 92.6%（含跨链衰减）。

### 3.3 intent.ts — IntentParser

**用途：** 将开发者传入的 `("swap", {protocol, tokenIn, amount})` 转换为结构化的 `IntentTemplate`。

**关键特性：**

- 支持 9 种 Action 类型
- Symbol → Address 自动解析（如 "USDC" → "0xA0b8..."）
- Decimal 精度自动转换（如 USDC 的 6 decimals → wei）
- 确定性 keccak256 hash
- 参数校验：非法 Action → UnknownActionError，缺字段 → ValidationError

**Agent 开发者的使用方式：**

```typescript
const intent = chiron.intent("swap", {
  protocol: "uniswap_v3",
  tokenIn: "USDC",
  tokenOut: "WETH",
  amount: "100",
});
```

### 3.4 decoder.ts — TxDecoder

**用途：** 将交易的原始 `calldata` 解码为人类可读的结构化参数。

**解码流程：**

```
target + data
    │
    ▼
selector 提取（前 4 bytes）
    │
    ├─ 命中已知表 → ABI 解码 → DecodedTx
    └─ 未知 → functionName = "unknown"（优雅降级）
    │
    ▼
delegatecall / CREATE2 检测
    │
    ▼
DecodedTx（target, selector, functionName, params, summary）
```

**内置的已知函数签名（预置 ABI）：**

```
exactInput, exactInputSingle, exactOutput, exactOutputSingle
swapExactTokensForTokens, swapExactETHForTokens
deposit, supply, withdraw, borrow, repay, swap
submit, exchange
```

### 3.5 checker.ts — ConsistencyChecker

**用途：** 执行 6 条确定性校验规则，判断交易是否与意图一致。

**6 条规则详解：**

| 规则 | 检查内容 | 判断标准 | 攻击防御 |
|---|---|---|---|
| R1 | 目标合约是否在协议表中 | `Registry.exists(target)` | 钓鱼合约 |
| R2 | 函数 Action 类型是否匹配 | `entry.actionType === intent.actionType` | Approve 替 Swap |
| R3 | TokenIn/Out 地址是否匹配 | `params.includes(intent.tokenIn)` | 代币替换 |
| R4 | 金额是否在 ±0.5% 滑点内 | `|amount - intent.amount| / intent.amount ≤ 0.5%` | 金额操纵 |
| R5 | 是否有意外参数 | receiver 匹配、无隐藏 Approve | 地址替换、隐藏授权 |
| R6 | 是否有隐藏 delegatecall/CREATE2 | `!data.includes(delegatecall)` | 代理执行 |

**输出结果：**

```typescript
{
  l1Result: 'PASS' | 'FAIL' | 'UNCERTAIN',
  details: CheckDetail[]       // 每条规则的详细结果
}
```

### 3.6 index.ts — Chiron 主入口

**用途：** 对外暴露的 SDK 接口。

```typescript
class Chiron {
  constructor(config: ChironConfig)
  
  // 创建意图
  intent(action: string, params: IntentParams): Promise<IntentTemplate>
  
  // L1 校验
  verify(intent: IntentTemplate, tx: TxCandidate): Promise<VerificationReceipt>
  
  // 异步存证（Phase 2 实装）
  storeReceipt(receipt: VerificationReceipt): Promise<void>
}
```

**3 行集成示例：**

```typescript
import { Chiron } from "@chiron/sdk";                // 第 1 行
const chiron = new Chiron({ chainId: 1 });            // 第 2 行
const r = await chiron.verify(intent, tx);            // 第 3 行
if (!r.allowed) throw new Error(`Blocked: ${r.reason}`);
```

## 六、已知局限

| 局限 | 影响 | 后续处理 |
|---|---|---|
| ethers 依赖安装在用户侧 | 用户需自行 `npm install` 后才能使用 | — |
| L1 不能覆盖 100% 场景 | 约 5% 的 UNCERTAIN 率不可避免 | Phase 4 L2 Verifier |
| 内置协议表需要维护更新 | 新协议需要 PR 合并后发版 | Phase 5 社区注册流程 |
| 无法检测 Intent 本身的恶意性 | 只比较 Intent 与 Tx 的一致性 | 设计上不解决（见 PRD）|
| receipt.ts 的 `src/receipt.ts` 中 `abi` 变量引用 | 需要在 `receipt.ts` 中添加被注释掉的 `const abi = ethers.AbiCoder.defaultAbiCoder();` 代码行 | Phase 2 修复 |
