## 二、合约架构

### 2.1 Chiron.sol — VerificationStore

**合约地址（示例）：** 部署后确定（≈ $50 Gas）

**核心存储模型：**

```solidity
mapping(bytes32 txHash → VerificationReceipt) public receipts;
mapping(address agent → bytes32[]) public agentReceipts;
mapping(address agent → uint256) public dailyNonce;
```

**VerificationReceipt 结构：**

```solidity
struct VerificationReceipt {
    bytes32 intentHash;      // 意图 hash
    bytes32 txHash;          // 交易 hash
    address agent;           // Agent 地址
    L1Result l1Result;       // PASS / FAIL / UNCERTAIN
    L2Result l2Result;       // NONE / CONSISTENT / INCONSISTENT（可选）
    uint256 timestamp;       // 写入时间
    uint256 blockNumber;     // 写入区块号
}
```

**对外接口：**

| 函数 | 权限 | 用途 |
|---|---|---|
| `storeReceipt(...)` | Anyone（有 Gas） | 存证一条校验结果 |
| `getReceipt(txHash)` | 公开读 | 查单条 |
| `getAgentReceipts(agent, offset, limit)` | 公开读 | 分页查 Agent 历史 |
| `getAgentStats(agent)` | 公开读 | 统计通过/拒绝/总数 |

### 2.2 核心工作流

```
Agent 层:
  生成意图 → 校验 → 签名 → 广播

链上存证（异步非阻塞）:
                                ┌─────────────────────┐
  After tx broadcast ──────────→│ storeReceipt()      │
                                │   txHash             │
                                │   intentHash         │
                                │   agent               │
                                │   l1Result            │
                                └─────────┬───────────┘
                                          │
             ┌────────────────────────────┼──────────────┐
             ▼                            ▼              ▼
       getReceipt()               getAgentReceipts()   getAgentStats()
       (按 txHash)                (分页查询)           (统计)
```

## 四、技术决策

### 4.1 异步存证设计

```
交易广播 ──────────→ 不等待 ──→ 继续下一步
         ↓
    异步 storeReceipt() ──→ 后台发送 → 等待交易收据
```

**为什么异步：**
- 存证不应该是交易的瓶颈（Agent 交易本身需要即时性）
- 如果存证失败，不阻断交易本身（链上数据仍可通过其他路径恢复）
- Gas 费用由 Agent 承担，异步可以批量合并存证

### 4.2 合约选择：单合约而非多合约

```
方案 A（选择）：单合约 VerificationStore 处理所有 Agent 的存证
方案 B（放弃）：每 Agent 部署独立存储合约

选择原因：
- 查询统一（一个地址就能查所有 Agent）
- 部署成本低（一次部署 ≈ $50 Gas，vs 每 Agent ≈ $50）
- 便于索引（The Graph 子图只需一个数据源）
```

### 4.3 合约的 on-chain data 存储 vs Event-only

```
方案 A（选择）：Mapping 存储 + Events
方案 B（放弃）：Only Events（靠 indexer 重建状态）

选择原因：
- 合约直接查询更快（不需要外部 indexer）
- getAgentStats() 需要聚合数据，Event-only 无法实现
- 但额外 Gas 成本：storeReceipt ≈ 30k Gas（含存储）
```
