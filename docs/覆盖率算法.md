# L1 协议注册表覆盖率算法

> 适用场景：汇报、技术评审、白皮书
> 核心公式一句话：**覆盖率 = Σ(已注册协议 TVL × 系数) / Σ(所有协议 TVL × 系数)**

---

## 一、算法定义

### 1.1 覆盖率公式

```
C = (Σ Registered(TVL_i × w_i × a_i)) / (Σ Total(TVL_i × w_i × a_i))

C     = L1 校验覆盖率（目标 ≥ 95%）
TVL_i = 协议 i 的锁仓价值（来源：DeFi Llama）
w_i   = Agent 活跃度权重（0.0 ~ 1.0）
a_i   = 跨链衰减系数（Source 链 = 1.0，非 Source 链 = 0.7）
```

### 1.2 权重标准

| 协议类型 | w_i | 原因 |
|---|---|---|
| DEX / Aggregator | 1.0 | Agent 最频繁的操作 |
| 借贷 (Lending) | 1.0 | Agent 自动再平衡的核心 |
| 流动性质押 (LST) | 1.0 | Agent 收益策略的基础 |
| 跨链桥 (Bridge) | 0.7 | Agent 跨链间或迁移时使用 |
| 收益聚合 | 0.7 | Agent 自动化策略的终点 |
| 稳定币 / CDP | 0.5 | Agent 使用但频率较低 |
| RWA / 保险 | 0.3 | Agent 目前较少涉及 |

### 1.3 计算过程（演示用）

```
Step 1: 从 DeFi Llama 获取全网 Top 50 协议的 TVL
Step 2: 按协议类型分配 Agent 活跃度权重 w_i
Step 3: 按链分配跨链衰减系数 a_i
Step 4: 计算 已注册协议 的加权 TVL
Step 5: 计算 全网加权 TVL
Step 6: C = Step 4 / Step 5
```

## 四、为什么这个算法是合理的

### 4.1 为什么用 TVL 而不是交易量

| 指标 | 优点 | 缺点 |
|---|---|---|
| **TVL** | 数据公开可验证、波动相对稳定、与 Agent 操作空间正相关 | 不直接反映 Agent 调用频率 |
| **交易量** | 反映实际使用频率 | 数据分散、易被刷量、波动剧烈 |
| **Gas 消耗** | 反映网络实际负载 | 不同协议 Gas 效率不同、跨链不可比 |

**结论：** TVL 是最稳定且 Agent 相关性最高的单一指标。交易量辅助验证。

### 4.2 为什么加 Agent 权重 w_i

Agent 不是均匀分布在所有协议上。一个 $100 亿的借贷协议（Agent 频繁存取）和一个 $100 亿的 RWA 协议（Agent 几乎不动）对 Agent 的意义完全不同。所以按类型加权，让权重反映 Agent 的行为模式。

### 4.3 为什么加跨链衰减 a_i

Agent 通常从一条 Source 链（Ethereum）发起操作，然后通过跨链桥扩展到 L2。L2 上的协议通常也同时存在于 Ethereum，Agent 在 Ethereum 上完成主要交互。所以 L2 协议对 Agent 的价值需要按比例衰减，避免重复计算和在 L2 上注册一些 Agent 几乎不用的协议。

## 五、局限性与改进方向

### 5.1 当前局限

| 局限 | 影响 | 缓解 |
|---|---|---|
| TVL 不直接等于 Agent 调用频率 | 可能高估或低估某些协议 | 用 w_i 权重修正 |
| Agent 行为数据不公开 | 无法精确计算真实覆盖率 | 用 TVL + 类型加权估算 |
| 跨链衰减系数 a_i 为经验值 | 缺乏数据支撑 | 后续用 on-chain 数据校准 |

### 5.2 后续校准方向

```
V1 (当前):  TVL × w_i × a_i                           ← 公开数据估算
V2:         TVL × w_i × a_i + OnChain_TxCount_Log     ← 加入链上交易量校准
V3:         Actual_Agent_Tx_Volume / Total_Tx_Volume   ← 需要 Agent 行为分析工具
```
