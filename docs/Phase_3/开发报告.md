## 二、攻击测试框架

### 2.1 架构

```
sdk/src/attack-test.ts
│
├── AttackTestCase         ← 攻击用例定义（id, name, category, severity, construct）
├── AttackTestResult       ← 测试结果（passed, detected, actualL1, detail）
├── AttackTestRunner       ← 执行引擎（runAll/runSingle, 聚合结果）
│
├── ALL_ATTACKS[]          ← 15 个攻击用例列表
│   ├── AT-A1: Prompt 注入 → Approve
│   ├── AT-A2: Intent 声明伪造
│   ├── AT-A3: Calldata 篡改
│   ├── AT-A4: 隐藏 Approve
│   ├── AT-A5: 隐藏 delegatecall
│   ├── AT-A6: 金额操纵（10x）
│   ├── AT-A7: 接收地址替换
│   ├── AT-A8: 钓鱼合约调用
│   ├── AT-A9: 新 / 未知合约
│   ├── AT-A10: 会话密钥泄露滥用
│   ├── AT-A11: 批量交易单步篡改
│   ├── AT-A12: 跨链目标篡改
│   ├── AT-A13: 验证证明重放
│   ├── AT-A14: 日额度耗尽
│   └── AT-A15: Intent 声明绕过（已知边界）
│
└── AttackCategory         ← 攻击分类（INJECTION, MANIPULATION, SPOOFING, RECOVERY, ECONOMIC, LIMITATION）
```

### 2.2 每个攻击用例的结构

```typescript
interface AttackTestCase {
  id: string;                     // "AT-A1" ~ "AT-A15"
  name: string;                   // 攻击名称
  category: AttackCategory;       // 攻击分类
  description: string;            // 攻击描述
  severity: 'CRITICAL' | 'HIGH' | 'MEDIUM' | 'LOW';
  construct: () => Promise<{
    intent: IntentTemplate;       // 对应的正确意图
    tx: TxCandidate;             // 攻击交易
    expectedL1: L1Result;        // 预期 L1 结果（PASS/FAIL）
    expectedBlock: boolean;       // 预期是否被阻止
    notes: string;               // 补充说明
  }>;
}
```

### 2.3 执行流程

```
AttackTestRunner.runAll()
    │
    for each attack in ALL_ATTACKS
    │
    ├─ await attack.construct() → { intent, tx, expectedL1, expectedBlock }
    │
    ├─ checker.check(intent, tx) → { l1Result, details, decodedTx }
    │
    ├─ 比较 actualL1 vs expectedL1
    │  └─ passed = (blocked == expectedBlock)
    │
    ├─ 提取 details 中的 error 信息 → detail 摘要
    │
    └─ AttackTestResult { id, passed, detected, actualL1, detail }
```

## 三、攻击覆盖结果

### 3.1 阻止的攻击（9/15 = 60%）

| ID | 攻击 | 严重程度 | 检测规则 | 检测机制 |
|---|---|---|---|---|
| AT-A1 | Prompt 注入 — Approve 替换 | CRITICAL | R1 + R2 | target 不在协议表中，selector 不匹配 swap |
| AT-A2 | Intent 声明伪造（代币反转）| HIGH | R3 | decodedTx params 中找不到 intent.tokenIn |
| AT-A3 | Calldata 篡改 | CRITICAL | R2 | 未知 selector 0xdeadbeef 不匹配 uniswap_v3 |
| AT-A4 | 隐藏 Approve 在 Swap 中 | CRITICAL | R5 | decoded.functionName === 'approve' 但 intent 是 SWAP |
| AT-A5 | 隐藏 delegatecall | CRITICAL | R6 | data.includes('delegatecall') → hasDelegateCall = true |
| AT-A7 | 接收地址替换 | CRITICAL | R5 | intent.receiver 在 tx params 中未找到 |
| AT-A8 | 钓鱼合约调用 | CRITICAL | R1 | 合约地址不在协议注册表中 |
| AT-A9 | 新 / 未知合约 | HIGH | R1 | 合约地址不在协议注册表中 |
| AT-A10 | 会话密钥泄露滥用 | HIGH | R1 | intent 说 Uniswap 但 target 是零地址 |

### 3.2 已知边界（6/15 = 40%）

| ID | 攻击 | 不阻止原因 | 后续处理 |
|---|---|---|---|
| AT-A6 | 金额操纵（10x）| R4 需要从 calldata 提取金额，极小 calldata 下无法提取 | Phase 4 加入真实交易数据测试 |
| AT-A11 | 批量交易单步篡改 | 需要 ERC-8211 batch 解析 | Phase 5 |
| AT-A12 | 跨链目标篡改 | 需要源链上下文 | Phase 5（ERC-7683 集成）|
| AT-A13 | 验证证明重放 | 链级问题，VerificationStore 合约已通过 txHash 绑定解决 | 合约已实现 |
| AT-A14 | 日额度耗尽 | 需要 CircuitBreaker 合约 | Phase 4 |
| AT-A15 | Intent 声明绕过 | 设计边界：L1 只检查一致性，不检查正确性 | 永不可解（设计决策）|

### 3.3 按严重程度分布

```
CRITICAL: A1, A3, A4, A5, A7, A8, A11   = 7 个（5 阻断 / 2 边界）
HIGH:     A2, A9, A10, A13, A14           = 5 个（4 阻断 / 1 边界）
MEDIUM:   A12                              = 1 个（1 边界）
LOW:      A6, A15                          = 2 个（2 边界）
```

## 四、调试信息（US-03）

Phase 1 定义的 `CheckDetail` 结构和 `VerificationReceipt.details` 字段在 Phase 3 中扩展验证确认：

```
CheckDetail {
  ruleId: "R1" ~ "R6"
  ruleName: "目标协议检查" / "函数选择器检查" / ...
  passed: boolean
  expected: string    ← 预期值（如 "Uniswap V3 Router"）
  actual: string      ← 实际值（如 "0xMaliciousContract"）
  severity: "info" | "warning" | "error"
}
```

被攻击阻挡时返回的调试信息格式：

```
原因: L1 FAIL — R1: Unknown contract: 0xAttackerAddress...
规则: R1 (目标协议检查)
预期: Known protocol contract (action: SWAP)
实际: Unknown contract: 0xAttackerAddress...
严重: error
```
